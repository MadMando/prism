Abstract

Retrieval-Augmented Generation (RAG) systems conventionally retrieve a flat ranked list of semantically similar passages, treating all retrieved knowledge as epistemically equivalent. This ignores the rich relational fabric of real corpora: some passages support a claim, others refute it, still others qualify or supersede it. We introduce PRISM (Propagation & Retrieval via Informed Semantic Mapping), a retrieval framework that layers a typed epistemic knowledge graph over an existing vector store and uses spreading activation with convergence scoring to retrieve knowledge structured by how it relates, not merely by how similar it is.

PRISM defines a ten-type epistemic edge taxonomy (SUPPORTS, REFUTES, SUPERSEDES, DERIVES_FROM, SPECIALIZES, CONTRASTS_WITH, IMPLEMENTS, GENERALIZES, EXEMPLIFIES, QUALIFIES), each assigned a propagation weight and an epistemic valence. At retrieval time, a query activates seed nodes via vector similarity; activation then propagates through the typed graph for h hops with per-type decay. Nodes reached by multiple independent seed paths receive a convergence bonus — operationalizing the insight from memory consolidation theory that knowledge accessible from many contexts is more reliably grounded. Results are returned not as a ranked list but as five structured epistemic buckets: PRIMARY, SUPPORTING, CONTRASTING, QUALIFYING, and SUPERSEDED.

PRISM requires no re-indexing of existing vector stores, degrades gracefully to pure vector retrieval when no graph is available, and ships as an open-source Python library (pip install prism-rag).

1. Introduction

Retrieval-Augmented Generation [Lewis et al., 2020] has become the dominant paradigm for grounding large language model (LLM) outputs in external knowledge. A standard RAG pipeline embeds a query, retrieves the top-K most similar passages from a vector store, and concatenates them as context for the generating model. The implicit assumption is that passages can be ranked by a single scalar similarity score — that relevance is ordinal.

This assumption breaks down in corpora that contain conflicting, temporally stratified, or structurally related knowledge. Consider a regulatory document corpus spanning ten years: the most recent standard may supersede an older one on the same topic; a practitioner guide may qualify a general principle with important exceptions; two competing frameworks may both be highly similar to a query while refuting each other. A flat ranked list delivers all of this to the LLM without structural annotation. The LLM must then implicitly reason about relationships that the retrieval layer discarded.

Our position: Retrieval is not a ranking problem. It is a knowledge structuring problem. The correct output of a retrieval step is not "the top-5 most similar passages" but rather "here is what supports the answer, here is what challenges it, here is what qualifies it, here is what has been superseded."

We introduce PRISM, a retrieval framework that operationalises this position through three components:

  1. Epistemic Graph Construction. An offline phase in which an LLM classifies relationships between semantically proximate passage pairs, producing a directed multigraph with typed, weighted edges.
  2. Typed Spreading Activation. A query-time activation algorithm inspired by Collins & Loftus's spreading activation theory [Collins & Loftus, 1975] in which activation propagates from vector-search seed nodes through typed edges, with per-type propagation weights that reflect the epistemic strength of each relationship.
  3. Convergence Scoring & Epistemic Bucketing. Nodes reached by multiple independent seed paths receive a convergence bonus; nodes are then assigned to one of five epistemic buckets based on the dominant valence of the edges through which they were reached.

PRISM makes no assumptions about the internal structure of the underlying vector store. The reference implementation ships adapters for LanceDB, ChromaDB, Qdrant, Weaviate, and pgvector behind a single VectorAdapter protocol, and works on top of any existing corpus without re-embedding or re-indexing. When no graph is available, it degrades gracefully to standard vector retrieval.

Contributions

  • A ten-type epistemic edge taxonomy with formal definitions, propagation weights, and valence assignments (§3.1).
  • A convergence-scoring variant of spreading activation that rewards nodes accessible from multiple independent query perspectives (§3.2).
  • An epistemic bucketing output schema that structures retrieval results for downstream LLM consumption (§3.3).
  • An open-source Python library (prism-rag v0.2.13) with built-in adapters for LanceDB, ChromaDB, Qdrant, Weaviate, and pgvector, diagnostic CLIs (prism-build, prism-stats, prism-inspect, prism-viz, prism-export), NetworkX and Neo4j export, a local interactive graph explorer (prism-explore), self-contained HTML graph export (prism-viz --format html), and a self-contained synthetic benchmark (pip install prism-rag).

2. Background and Related Work

2.1 Retrieval-Augmented Generation

RAG [Lewis et al., 2020] augments an LLM with a retrieval component, conditioning generation on retrieved passages rather than relying solely on parametric knowledge. Dense retrieval using bi-encoder models [Karpukhin et al., 2020] has largely supplanted sparse BM25 retrieval for semantic similarity. Iterative retrieval methods such as IRCoT [Trivedi et al., 2022] interleave retrieval and reasoning, improving multi-hop question answering at the cost of latency. PRISM targets single-step retrieval but provides richer structural context than flat dense retrieval.

2.2 Graph-Augmented RAG

Microsoft GraphRAG [Edge et al., 2024] extracts entity and community structures from a corpus and uses them to support both local (entity-centric) and global (summary-centric) query modes. LightRAG [Guo et al., 2024] improves efficiency with a dual-level retrieval architecture over entity-relationship graphs. Neither system assigns epistemic types to edges; relationships are treated as generic "relates to" links with no propagation semantics.

2.3 HippoRAG

The most architecturally proximate prior work is HippoRAG [Gutiérrez et al., 2024], which draws on the hippocampal indexing theory of human long-term memory [Teyler & DiScenna, 1986] to motivate a retrieval architecture combining LLM-extracted knowledge graphs with Personalized PageRank (PPR). HippoRAG demonstrates substantial gains on multi-hop QA benchmarks — up to 20% over state-of-the-art — and shows that single-step graph-augmented retrieval can match iterative retrieval at a fraction of the cost.

PRISM shares HippoRAG's neurobiological motivation but differs in three critical ways. First, HippoRAG's graph is built from entity co-occurrence (named entity pairs extracted from passages); PRISM's graph encodes epistemic relationships between passages with a typed taxonomy. Second, HippoRAG uses Personalized PageRank, which assigns a global importance score per node; PRISM uses spreading activation, which is source-aware — it tracks which seed nodes independently activate which graph nodes, enabling convergence scoring. Third, HippoRAG returns a ranked list; PRISM returns structured epistemic buckets.

2.4 Spreading Activation Theory

Collins & Loftus [1975] proposed spreading activation as a model of human semantic memory: concepts are nodes in a network, associations are weighted edges, and activation propagates from cued concepts to related ones with decay proportional to path length. PRISM directly instantiates this model with a query-derived initial activation distribution, per-type propagation weights, and a hop-based decay schedule.

2.5 Knowledge Representation and Epistemic Logic

The PRISM edge taxonomy draws on work in epistemic logic [Fagin et al., 2003] and knowledge representation [Brachman & Levesque, 1985]. The distinction between positive, dialectical, qualifying, and temporal edge valences echoes the defeasible reasoning literature [Pollock, 1995], in which arguments can be undercut (qualified), rebutted (refuted), or defeated (superseded).

3. The PRISM Framework

3.0 Problem Formulation

Let C = {p1, p2, …, pm} be a corpus of text passages, each associated with a source document src(pi). Let E : C → ℝd be an embedding function. Given a query q, standard dense retrieval returns top-K{E(q) · E(pi)}.

PRISM augments this with an epistemic graph G = (C, ℰ) where each edge eij = (pi, pj, r, c) carries a relation type r ∈ ℛ and a confidence score c ∈ [0,1]. Retrieval returns a structured result 𝒪 = {𝒷primary, 𝒷supporting, 𝒷contrasting, 𝒷qualifying, 𝒷superseded}.

3.1 Epistemic Edge Taxonomy

We define a ten-type relation taxonomy ℛ, shown in Table 1. Each type is assigned a propagation weight w(r) ∈ (0,1] controlling how strongly activation flows across edges of that type, and an epistemic valence v(r) ∈ {positive, qualifying, dialectical, temporal} determining bucket assignment.

Table 1: Epistemic edge taxonomy with propagation weights and valences.
TypeValenceWeightSemantics
SUPPORTS positive 0.90pi provides evidence reinforcing pj
DERIVES_FROM positive 0.85pi is logically derived from pj
SPECIALIZES positive 0.80pi is a specific instance of pj
IMPLEMENTS positive 0.75pi concretely realizes the concept in pj
EXEMPLIFIES positive 0.75pi is a concrete example illustrating pj
GENERALIZES positive 0.70pi is a broader abstraction of pj
QUALIFIES qualifying 0.65pi adds conditions or exceptions to pj
CONTRASTS_WITHdialectical 0.55pi and pj take different positions
REFUTES dialectical 0.50pi directly contradicts pj

The weight ordering reflects epistemic strength: SUPPORTS carries activation most strongly because it represents corroborating evidence; SUPERSEDES carries it most weakly because historical context, while relevant, is less actionable than current material.

3.2 Graph Construction

Graph construction is an offline phase executed once per corpus.

Step 1: Candidate pair generation. For each passage pi, we retrieve its k-nearest neighbors by cosine similarity over the embedded corpus. By default, extraction spans all passage pairs (cross_source_only=False), including intra-document pairs. Early experiments restricted extraction to cross-source pairs (src(pi) ≠ src(pj)), which reduced candidate volume roughly 3–4× but left 51% of corpus passages with no graph edges — producing a graph too sparse for spreading activation to improve upon plain vector search. Including intra-document pairs increases candidate volume but yields substantially richer graphs (2–3× more edges on a 30k-chunk corpus), enabling spreading activation to fire on most queries rather than silently falling back to vector retrieval. Intra-document epistemic relationships — particularly EXEMPLIFIES and SPECIALIZES — are common and meaningful: a later passage that illustrates a concept defined earlier in the same document is epistemically connected even if it comes from the same source.

Step 2: LLM-based relation extraction. Candidate pairs are batched (default: 20 per call) and submitted concurrently to an OpenAI-compatible LLM with a structured prompt requesting JSON output specifying relation type, direction, and confidence (20 concurrent requests via async I/O). Pairs with confidence c < θ (default θ = 0.65) are discarded. An optional Stage 1 fast-model pre-filter (binary yes/no via Ollama) can halve API cost when a local, low-latency model (<5 GB) is available; remote Ollama instances with GPU VRAM constraints cannot run true parallel inference and the filter offers no throughput benefit in that setting.

Step 3: Graph serialization. The completed graph is serialized to gzip-compressed JSON. At retrieval time it loads into memory in a single pass, sub-second for graphs up to 100k nodes.

3.3 Typed Spreading Activation

Seed initialization. The top-Ks passages by vector similarity are designated seeds:

a(0)(pi) = cos(E(q), E(pi))    if pi ∈ seeds,    else 0

Propagation. For h hops, activation propagates from each active node to its graph neighbors, weighted by edge type and attenuated by a global decay factor δ:

a(t)(pj) = Σ(pi, pj, r) ∈ ℰ a(t−1)(pi) · w(r) · cij · δt

where w(r) is the propagation weight for edge type r and cij is the LLM-assigned confidence for that edge (stored as a single combined weight in the serialized graph). Activation accumulates (sum-pool) across all incoming neighbors, so nodes corroborated by multiple paths receive proportionally higher scores. Propagation also traverses edges in the reverse direction, scaled by a penalty of 0.6, allowing downstream passages to activate their upstream sources at reduced strength.

Convergence tracking. Throughout propagation, we track the set of seed nodes that have contributed activation to each node. The convergence score is the fraction of seeds that independently activated node pj:

conv(pj) = |seeds(pj)| / |seeds|

Final scoring. The final score combines activation magnitude with convergence:

f(pj) = a(pj) · (1 + λ · conv(pj))

where λ is the convergence weight (default λ = 0.4). This multiplicative formulation preserves the full magnitude of activation while applying a convergence bonus: a node reached by every seed is scaled by up to 1 + λ = 1.4, whereas a node activated by only one seed receives no bonus. Nodes with high convergence — corroborated by multiple independent query perspectives — are thus promoted without suppressing strongly-activated unique matches.

3.4 Epistemic Bucketing

Retrieved nodes are assigned to five epistemic buckets based on seed status and dominant edge valence:

  1. PRIMARY: Seed nodes and non-seed nodes with convergence ≥ τ. Core, convergently supported answer passages.
  2. SUPPORTING: Non-seed nodes reached primarily via positive valence edges (SUPPORTS, DERIVES_FROM, GENERALIZES, SPECIALIZES, IMPLEMENTS, EXEMPLIFIES).
  3. CONTRASTING: Non-seed nodes reached primarily via dialectical valence edges (REFUTES, CONTRASTS_WITH).
  4. QUALIFYING: Non-seed nodes reached via QUALIFIES edges.
  5. SUPERSEDED: Non-seed nodes reached via SUPERSEDES edges — historically relevant but now-replaced content.

An example of the structured output format delivered to the LLM:

## PRIMARY
[1] source-a  p.14  §2.1  (score: 0.923)
    The core relevant passage from your corpus...

## CONTRASTING VIEWS
[1] source-b  p.67  §5.0  (score: 0.812  [via: refutes])
    A passage that directly challenges the above...

## QUALIFICATIONS & NUANCES
[1] source-c  p.38  §3.1  (score: 0.712  [via: qualifies])
    Conditions and exceptions to consider...

─ 1 primary · 1 supporting · 1 contrasting · 1 qualifying · 0 superseded ─

Edge typing vs. entity co-occurrence. HippoRAG builds its graph by extracting named entity mentions and recording co-occurrence within passages. This captures topical proximity but not epistemic relationship. Two passages that both mention the same concept are linked regardless of whether one supports, refutes, or qualifies the other. PRISM's extraction step explicitly classifies the epistemic type of each relationship, producing a graph whose edge types carry semantics usable both for activation weighting and for bucket assignment.

Spreading activation vs. Personalized PageRank. PageRank and spreading activation are both diffusion processes on graphs, but they differ in a critical way: PageRank is global — it assigns a single importance score that converges to the stationary distribution of a random walk, marginalizing over all starting points. Spreading activation is source-aware — each seed's contribution is tracked independently throughout propagation. This source-awareness enables convergence scoring: we can ask not just "how activated is this node?" but "how many independent perspectives on the query converge here?"

Convergence scoring as a reliability signal. The convergence score conv(pj) measures the fraction of independent query-relevant seeds that transitively activate node pj. A passage activated by five independent seeds is more likely to represent genuinely relevant, cross-validated knowledge than one activated by a single strong seed path. This is directly analogous to memory consolidation in hippocampal indexing theory: memories cued from multiple contexts become more stable and accessible [McClelland et al., 1995].

5. Preliminary Evaluation

5.1 Synthetic Benchmark

To demonstrate epistemic bucketing without requiring a labelled QA dataset, we construct a synthetic benchmark consisting of 8 passages from 3 fictional sources concerning a shared topic (accountability in distributed systems). The corpus is designed with known ground-truth epistemic relationships: one passage REFUTES another, one SUPERSEDES an earlier treatment, one QUALIFIES a general principle. A small epistemic graph of 6 edges is constructed directly (without LLM extraction) for controlled evaluation.

Standard vector retrieval result:

Query: "accountability mechanisms in distributed systems"

[1] framework-a: "Accountability requires clear ownership..."   (0.912)
[2] framework-b: "Distributed accountability is incompatible..." (0.891)
[3] standard-c:  "Accountability must be assigned at..."         (0.856)
[4] framework-a: "Historical note: early frameworks treated..."  (0.821)
[5] framework-b: "An exception applies when..."                  (0.798)

PRISM epistemic retrieval result:

Query: "accountability mechanisms in distributed systems"

## PRIMARY
[1] framework-a: "Accountability requires clear ownership..."   (0.912)
[2] standard-c:  "Accountability must be assigned at..."        (0.856)

## CONTRASTING VIEWS
[1] framework-b: "Distributed accountability is incompatible..." (0.891  [via: refutes])

## QUALIFICATIONS & NUANCES
[1] framework-b: "An exception applies when..."                  (0.798  [via: qualifies])

## SUPERSEDED (historical context only)
[1] framework-a: "Historical note: early frameworks treated..."  (0.821  [via: supersedes])

The epistemic structure transforms a flat list into an answer that explicitly flags the contradiction (REFUTES), the conditions (QUALIFIES), and the historical context (SUPERSEDES). An LLM consuming the second format has the information it needs to produce a calibrated, nuanced answer; one consuming the first must discover these relationships implicitly or miss them entirely.

5.3 Edge Type Distribution

We examined a random sample of 50 extracted edges from the Build 2 corpus and classified them as correct, incorrect, or marginal by manual annotation. Of these, 39 (78%) were judged correct, 7 (14%) marginal (relationship exists but type assignment debatable), and 4 (8%) incorrect. The most common error was conflating QUALIFIES with CONTRASTS_WITH when a passage adds significant exceptions to another.

Full edge type distribution across all 9,989 edges in Build 2 is shown in Figure 1. EXEMPLIFIES dominates at 41.8% (4,176 edges), reflecting that intra-document passages most often stand in an illustrative relationship — a later passage exemplifies or instantiates a concept introduced earlier in the same document. CONTRASTS_WITH at 13.7% (1,369) captures competing frameworks and different definitional positions across the multi-source corpus. Hard contradiction (REFUTES) is rare at 0.3% (27), consistent with the governance domain where sources differ in emphasis rather than making mutually exclusive factual claims. The predominance of positive-valence edges (EXEMPLIFIES + SPECIALIZES + IMPLEMENTS + GENERALIZES + DERIVES_FROM + SUPPORTS = 80.3%) means the SUPPORTING bucket activates frequently, while CONTRASTING and SUPERSEDED serve as useful minority signals when present.

5.4 Graph Traversal Depth: Cross-Source vs. All-Source

A direct test of whether a denser graph improves retrieval is to compare how far spreading activation propagates beyond the initial seed set. We issued the same governance query — "what is data stewardship and what governs it" — to both builds and logged seed count and activated node count.

Table 4: Spreading activation traversal depth comparison — same query, same corpus, different graph density.
Build Edges Seeds (vector) Nodes activated Graph expansion
Build 1 (cross-source only) 3,571 19 19 +0 (no traversal)
Build 2 (all pairs) 9,989 19 34 +15 (+79%)

With Build 1, spreading activation found no edges to traverse from the 19 seed nodes — activated nodes equalled seeds, and the retrieval result was identical to plain vector search. With Build 2's denser graph, the same 19 seeds activated 15 additional non-seed nodes through typed edge traversal, surfacing passages that would have ranked outside the top-K by similarity alone. The expanded node set populates the SUPPORTING, QUALIFYING, and CONTRASTING buckets with content that vector similarity would not have retrieved.

This illustrates the critical sensitivity of spreading activation to graph density: below a connectivity threshold, the algorithm degrades silently to vector search. For a 30k-passage corpus, approximately 0.64 edges/node (Build 2) is sufficient for reliable traversal; 0.23 edges/node (Build 1) is not. We recommend monitoring average edge density after building and rebuilding with cross_source_only=False if spreading activation rarely fires.

6. Planned Evaluation

Multi-hop QA benchmarks. We will evaluate PRISM against dense retrieval and HippoRAG [Gutiérrez et al., 2024] on HotpotQA [Yang et al., 2018], MuSiQue [Trivedi et al., 2022], and 2WikiMultiHopQA [Ho et al., 2020]. These benchmarks require connecting information across documents — exactly the setting where graph-augmented retrieval demonstrates gains. Primary metrics: Recall@K and Answer F1.

Epistemic structure quality. We will construct a synthetic adversarial QA set with 200 questions over a corpus with deliberately planted contradictions, supersession events, and multi-source convergence. Ground truth specifies not just the correct answer but the correct epistemic role of each supporting passage. This evaluates what existing benchmarks cannot: whether PRISM correctly identifies the epistemic role of retrieved passages, not just their relevance.

LLM reasoning quality study. A human evaluation comparing LLM answers generated from (a) flat top-5 vector context and (b) PRISM structured context, on questions where the corpus contains conflicting information. The hypothesis is that LLMs given PRISM's bucketed output will produce fewer factual errors and more appropriately hedged answers on contradiction-containing queries.

7. Limitations

Graph quality is LLM-dependent. The epistemic graph is produced by an LLM that can misclassify relationships. Graph quality is correlated with chunk quality, chunking strategy, and LLM capability. We recommend reviewing extracted edges before trusting them in high-stakes applications.

Confidence scores are not calibrated probabilities. Confidence values returned by the extraction LLM are self-reported estimates, not calibrated. They are used as a filter and as edge weights but should not be treated as precise measures of relationship strength.

Noisy edges degrade retrieval. If the graph contains many false-positive edges, spreading activation propagates to irrelevant nodes. In the worst case, PRISM performs worse than pure vector retrieval. Monitoring edge yield rate and type distribution after building is recommended.

Candidate-pair scope is critical to graph density. Early experiments with cross-source-only pair generation left 51% of corpus passages with no graph edges, producing a graph where spreading activation never traversed beyond seed nodes — the system silently fell back to plain vector search on every query. Restricting to cross-source pairs is counterproductive on single-source corpora and harmful on multi-source corpora where intra-document epistemic structure is rich. The library now defaults to all-pair extraction (cross_source_only=False). Practitioners should monitor average edges per node after building; values below ~0.4 suggest the graph is too sparse to add value.

Stage 1 filter viability is hardware-dependent. The optional Stage 1 Ollama pre-filter assumes low per-call latency. Remote Ollama instances with GPU VRAM constraints cannot run true parallel inference; even with OLLAMA_NUM_PARALLEL=20, binary classification calls execute serially (~1.5 s/pair), making the filter slower than skipping it entirely. Users with local Ollama instances and spare VRAM headroom will benefit (50% pair reduction, ~half the API cost); users with remote or memory-constrained Ollama should use --no-filter and rely on Stage 2 alone.

Graph construction cost. The one-time build requires thousands of LLM API calls (hours for large corpora) and has no incremental update path. Adding new documents currently requires a full rebuild. On a 30k-passage corpus with cross_source_only=False, Stage 2 async classification takes approximately 70 minutes with 20 concurrent requests at ~1.5 s/batch.

No QA evaluation benchmark yet. Retrieval quality improvements are supported by qualitative analysis and synthetic benchmarks only. Head-to-head comparison on standard QA datasets is in preparation (§6).

8. Conclusion

We have presented PRISM, a retrieval framework that augments vector search with a typed epistemic knowledge graph and spreading activation to produce structured, epistemically annotated retrieval results. The core insight is that knowledge retrieval should return not a ranked list of similar passages but a structured account of how retrieved knowledge relates to the query: what supports it, what challenges it, what qualifies it, what supersedes it.

PRISM operationalises this through three novel components: a ten-type epistemic edge taxonomy with formal propagation semantics; a convergence-scoring variant of spreading activation that rewards multi-perspective consensus; and an epistemic bucketing output schema that structures retrieval results for LLM consumption. The system requires no re-indexing of existing vector stores and degrades gracefully to standard vector retrieval when no graph is available.

The framework is available as an open-source Python package at pypi.org/project/prism-rag/ and github.com/MadMando/prism. We invite the community to evaluate PRISM on their own corpora, contribute edge type extensions, and collaborate on the quantitative benchmarks described in §6.
